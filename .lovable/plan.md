
# План: «Живой Блокнот» → продакшн

Делаем фазами. После каждой фазы — проверка, что всё работает, и только потом следующая. Существующие данные стираем (по вашему выбору), стартуем чисто с авторизацией.

---

## Фаза 1 — Авторизация и безопасность БД

**Цель:** никто кроме вас не видит ваши данные.

1. Включить Email/Password + Google OAuth (через Lovable брокер).
2. Создать страницу `/auth` (вход + регистрация, переключение табами).
3. Защитить все рабочие маршруты слоем `_authenticated/` (главная, расписание, ученики, финансы, посещаемость, ДЗ, аналитика, ИИ, настройки). Незалогиненных редиректит на `/auth`.
4. Миграция БД:
   - Очистить все таблицы.
   - Добавить `owner_id uuid NOT NULL` во все 8 бизнес-таблиц.
   - Заменить политики `public all` на политики `auth.uid() = owner_id`.
   - Добавить триггер автозаполнения `owner_id` при INSERT.
   - Добавить FK на `students(id) ON DELETE CASCADE` для `finance`, `attendance`, `homework`, `schedule_slots`, `lessons_conducted`.
   - Индексы по `owner_id` и `student_id`.
5. Все запросы в коде переписать через `createServerFn` + `requireSupabaseAuth` (вместо прямых вызовов supabase из браузера) — это правильный паттерн для TanStack Start и закрывает дыру с anon-ключом.
6. Подписка на `onAuthStateChange` в `__root.tsx` для инвалидации кеша.

**Проверка:** регистрируетесь, добавляете ученика — видите только своего.

## Фаза 2 — Таблица `lessons` и календарь

**Цель:** настоящий календарь с историей и статусами уроков.

1. Создать таблицу `lessons`:
   - `id`, `owner_id`, `student_id` (FK), `scheduled_date`, `scheduled_time`, `duration_min`, `status` (`planned|completed|cancelled|moved`), `notes`, `created_at`, `updated_at`.
   - RLS по `owner_id`, индексы по дате и ученику.
2. Сгенерировать историю: для каждого `schedule_slot` создать инстансы уроков за последние 3 месяца + на 1 месяц вперёд. Статусы вывести из существующего `attendance` (present → completed, absent → cancelled).
3. Календарь на главной и в расписании: показывает не шаблоны, а конкретные инстансы из `lessons`.
4. Действия по уроку: отметить проведённым, перенести (drag-drop или модалка с датой), отменить, добавить заметку.
5. `lessons_conducted` оставляем только для обратной совместимости аналитики, или мигрируем счётчики из `lessons` (выбор в момент работы).

**Проверка:** уроки видны в календаре, можно отмечать, переносить, видеть историю.

## Фаза 3 — Бэкапы, поиск, soft delete, валидация

### 3.1. Бэкапы (в Настройках)
- Экспорт CSV: каждая таблица отдельным файлом.
- Экспорт JSON: один файл со всеми данными пользователя.
- Импорт JSON: загрузить ранее экспортированный бэкап (с проверкой структуры, owner_id перезаписывается на текущего пользователя).

### 3.2. Поиск и фильтры в «Ученики»
- Строка поиска по имени.
- Фильтры: предмет, есть долг, есть урок сегодня/на этой неделе.
- Сортировка: по имени, по дате добавления, по сумме долга.

### 3.3. Soft delete везде
- `deleted_at timestamptz` во всех 8 таблицах.
- Все запросы фильтруют `deleted_at IS NULL`.
- Раздел «Корзина» в Настройках: список удалённого, кнопки «Восстановить» и «Удалить навсегда».
- Удаление ученика мягко каскадит (помечает связанные записи), восстановление возвращает.

### 3.4. Настройки уведомлений
- В Настройках добавить переключатели: напоминания об уроках, о платежах, о ДЗ.
- Хранить в новой таблице `user_preferences` (по `owner_id`).
- Существующий cron `/api/public/hooks/lesson-reminders` уважает эти флаги.

### 3.5. UX и валидация
- Zod-схемы на формах (имя ученика, телефон, суммы, время).
- Серверная валидация в `createServerFn` теми же схемами.
- Skeleton-загрузки в списках вместо пустоты.
- Понятные empty-states («Пока нет учеников — добавьте первого»).
- Toast-ошибки с человеческим текстом.

## Технические детали (для справки)

### Стек серверной логики
Всё — через `createServerFn` с `requireSupabaseAuth` middleware. Прямые `supabase.from(...)` из компонентов остаются только для realtime/auth-флоу. Бэкенд-маршруты под `/api/public/*` — только для cron-вебхуков.

### Стратегия миграций
- Каждая фаза = одна миграция (или 2-3 связанные).
- Все миграции идемпотентны (IF NOT EXISTS, DROP POLICY IF EXISTS).
- Откат: храним rollback-SQL в комментарии к каждой миграции, по запросу применяем.

### Что точно не трогаем
- Визуальный стиль (liquid-glass, дизайн-токены, шрифты).
- BottomNav и навигацию.
- Структуру страниц и компонентов UI.
- Существующий AI-ассистент и push-инфраструктуру (только добавим owner_id).

### Что может сломаться и как защищаемся
- Старые данные стираются → согласовано.
- AI-ассистент после Фазы 1 будет видеть только данные текущего пользователя → правильное поведение.
- `lessons_conducted` счётчик после Фазы 2 может расходиться с `lessons` → решим (заменить виджет аналитики на запрос к `lessons`).

## Порядок выполнения

```text
Фаза 1 (auth + RLS)          → проверка → коммит
   ↓
Фаза 2 (lessons + календарь) → проверка → коммит
   ↓
Фаза 3 (backup/поиск/trash)  → проверка → готово
```

Начнём с **Фазы 1**. После её завершения — пауза, вы тестируете, потом переходим к Фазе 2.
